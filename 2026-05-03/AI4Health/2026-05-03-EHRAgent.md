# EHRAgent：让 LLM 像数据分析师一样处理电子病历

## 基本信息

- **论文标题**：EHRAgent: Code Empowers Large Language Models for Few-shot Complex Tabular Reasoning on Electronic Health Records
- **论文链接**：https://arxiv.org/abs/2401.07128
- **发表时间**：2024年1月（EMNLP 2024 收录）
- **机构**：加州大学圣地亚哥分校（UCSD）

---

## 背景知识：电子健康记录（EHR）是什么？

**EHR（Electronic Health Records，电子健康记录）**：
医院里记录患者所有诊疗信息的数字系统，包含：
- **结构化数据**：检查值（血压、血糖、肌酐……）、诊断代码（ICD codes）、用药记录
- **非结构化数据**：医生书写的病程记录、出院小结
- **时序数据**：每次就诊的时间、检查时间点……

EHR 数据是一张**超级复杂的大表格**，时间跨度可能达数年，每个患者有数百到数千条记录。

**临床数据问答的挑战**：
医生或研究者想问："在我的患者群中，同时有高血压和糖尿病、且近90天内住过院的患者有多少？他们的平均住院时长是多少？" 这类问题：
- 需要跨多个表、多个时间窗口进行复杂查询
- 传统方法：需要会写 SQL 或 Python，大多数临床医生不会
- 直接问 LLM：LLM 容易"直接编造"一个数字而不是真正查询数据

**什么是 AI Agent（智能体）？**
- 不只是"一问一答"的 LLM，而是能够**使用工具、执行多步骤行动**的 AI
- 就像一个助手，给它一个目标，它会自己想步骤、调用工具、检查结果、调整策略

---

## 这篇论文干了什么？

EHRAgent 让 LLM 变成一个**能自主写代码分析 EHR 的智能体**：
- 用户用自然语言提问（无需懂 SQL/Python）
- EHRAgent 自动将问题转化为可执行的 Python 代码
- 执行代码、查询 EHR 数据库
- 根据执行结果给出准确答案

---

## 怎么干的？

### 核心思路："让 LLM 写代码，而不是直接回答"

传统做法（直接问答）：
```
问：ICU 中患者平均住院天数是多少？
LLM 答：大约7天（直接编造，可能错误）
```

EHRAgent 的做法（代码驱动）：
```
问：ICU 中患者平均住院天数是多少？
LLM 生成 Python：
  icu_stays = df[df['unit']=='ICU']
  avg_days = icu_stays['length_of_stay'].mean()
  print(avg_days)
执行代码 → 得到真实数字 → 回答：6.3天
```

### 系统架构

**四个关键模块**：

1. **意图理解（Intent Parsing）**：
   - 理解自然语言问题的意图
   - 识别涉及哪些数据表、哪些时间范围、什么逻辑运算

2. **代码生成（Code Generation）**：
   - 把意图转化为 Python pandas 代码
   - LLM 基于 EHR 数据库的 schema（表结构说明）生成代码

3. **代码执行与调试（Execution & Debugging）**：
   - 自动执行生成的代码
   - 如果报错，LLM 看错误信息后自动修改代码重试（最多3次）
   - 这是"智能体"行为的核心：看反馈、调整、再试

4. **答案提取（Answer Extraction）**：
   - 从代码执行结果中提取并整理最终答案
   - 用自然语言表达给用户

### 少样本学习（Few-shot）

EHR 数据库结构复杂，需要教 LLM 如何使用：
- 提供少量（1-5个）例子展示如何把自然语言转为代码
- 这些例子作为"提示词"的一部分
- LLM 模仿例子的风格生成新代码

### 评估数据集

在两个公开 EHR 数据集上测试：
- **MIMIC-III**：美国 ICU 患者数据（40,000+ 患者）
- **eICU**：多中心 ICU 数据（200,000+ 患者）

---

## 实验结果与结论

- 与直接问答 LLM 相比，EHRAgent 在复杂 EHR 问答任务上准确率提升约 **20-40%**
- 特别在需要多步骤推理（JOIN 多个表、时间窗口过滤）的问题上优势明显
- 自动调试机制能修复约 60% 的代码错误
- 相比专门训练的 EHR 问答模型，在**少样本设置**下更灵活

**关键结论**：让 LLM 写代码而非直接回答，是处理结构化医疗数据查询的有效范式；"代码执行 + 自动调试"的 Agent 框架显著减少幻觉。

---

## 局限性（Limitations）

1. **代码安全性**：自动生成和执行代码存在安全风险（如果数据库有写权限可能导致数据损坏）

2. **表结构依赖**：需要预先向 LLM 提供准确的数据库 schema，不同医院 EHR 系统结构差异大

3. **非结构化数据处理弱**：对自由文本的病程记录处理能力有限

4. **调试次数有限**：复杂错误可能在3次重试内无法修复

5. **数据库规模限制**：当 EHR 表格数据量极大时，代码执行效率可能成为瓶颈

---

## 为什么值得读？

EHRAgent 展示了 **"LLM + 代码执行"** 这个通用 Agent 范式在医疗领域的应用。这个思路不仅适用于 EHR，也可以推广到任何需要精确数据查询的医疗场景。如果你对 AI Agent、临床 NLP 或医疗数据科学感兴趣，这是一篇思路清晰的好论文。
