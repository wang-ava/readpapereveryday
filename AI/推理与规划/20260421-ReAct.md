# ReAct：让AI像人一样"想一想、查一查、再回答"

## 基本信息

- **论文标题**：ReAct: Synergizing Reasoning and Acting in Language Models
- **作者**：Shunyu Yao 等，普林斯顿大学 / Google Research
- **发表会议**：ICLR 2023
- **论文链接**：https://arxiv.org/abs/2210.03629
- **代码**：https://github.com/ykwd/react

---

## 背景知识：为什么 AI 需要"工具"？

### 语言模型的两大局限

即使有了思维链（CoT）提示，大语言模型仍然面临两个根本局限：

**局限1：知识截止日期**
模型的知识停留在训练数据结束的那一天，无法获取最新信息。
> 问"今天的股价是多少"，模型只能乱猜

**局限2：幻觉（Hallucination）**
模型有时会自信地捏造不存在的事实，无法区分"我真的知道"和"我在瞎编"。

**解决思路**：给语言模型配备**工具**（搜索引擎、计算器、数据库……），让它在推理过程中随时"查资料"。

### Agent（代理）的概念

**Agent** 是指能自主感知环境、做决策、执行行动的AI系统。

语言模型 + 工具调用能力 = AI Agent

一个基础的 Agent 循环：
1. 观察当前情况（Observation）
2. 推理应该做什么（Reasoning）
3. 采取行动（Action，如搜索、计算）
4. 获得新的观察结果，回到步骤1

### 之前的方法：推理和行动是分离的

**CoT（思维链）**：
- 优势：强大的多步推理
- 缺点：推理过程中不能查资料，知识全靠"背诵"，容易幻觉

**Action-only 方法（只使用工具）**：
- 优势：信息准确，来自真实查询
- 缺点：没有推理能力，无法处理复杂的多跳问题（需要整合多个来源的信息）

ReAct 的思路：**把推理（Reasoning）和行动（Acting）交织在一起**，两者互相增强。

---

## 这篇论文干了什么？

ReAct 提出了一种提示框架，让语言模型按照 **思考（Thought）→ 行动（Action）→ 观察（Observation）** 的循环来完成复杂任务：

1. 模型先"想一想"：分析当前情况，决定下一步该做什么
2. 模型执行一个工具调用（如搜索维基百科）
3. 模型看到工具返回的结果（Observation）
4. 重复上述步骤，直到得出答案

在知识问答、事实验证、任务规划等任务上，ReAct 显著优于单纯推理（CoT）和单纯行动（Action-only）的方法。

---

## 怎么做到的？

### 核心：交织推理和行动的提示格式

ReAct 使用了非常简单的 few-shot 提示，关键在于**示例中展示了"思考-行动-观察"交织的格式**：

**示例**（简化版，知识问答任务）：
```
问题：科罗拉多臭虫破坏哪种植物，这种植物最早种植于哪个州？

思考：我需要先搜索科罗拉多臭虫破坏什么植物。
行动：搜索[科罗拉多臭虫]
观察：科罗拉多马铃薯甲虫（学名：Colorado potato beetle）...主要危害马铃薯...

思考：找到了，是马铃薯。现在我需要搜索马铃薯最早在哪个州种植。
行动：搜索[马铃薯 美国 最早种植]
观察：马铃薯在18世纪早期从欧洲传入美国，最早在新罕布什尔州（New Hampshire）大规模种植...

思考：根据两次搜索的结果，答案是新罕布什尔州。
最终答案：新罕布什尔州
```

这个格式中，每个"思考"步骤都是模型自己生成的自然语言推理，每个"行动"步骤是调用外部工具（这里是维基百科搜索），每个"观察"步骤是工具返回的真实结果。

### 可用的行动类型（取决于任务）

**知识问答（HotpotQA, FEVER 等）**：
- `搜索[实体名]`：在维基百科中搜索该实体，返回段落
- `查找[关键词]`：在搜索结果中找特定词语
- `完成[答案]`：给出最终答案

**决策任务（ALFWorld 文本游戏）**：
- 各种游戏行动：`去厨房`、`拿起苹果`、`打开冰箱`……
- 思考则帮助规划："我需要找一个温热的苹果，可能在锅里或微波炉里"

**购物网站任务（WebShop）**：
- `搜索[商品]`
- `点击[按钮/链接]`

### 为什么推理和行动要交织？

- **推理帮助行动**：推理步骤让模型知道下一步该搜什么、该做什么，避免盲目尝试
- **行动帮助推理**：外部工具返回的信息更新了模型的"知识状态"，使后续推理基于真实信息而非记忆，减少幻觉
- **更可解释**：思考步骤是自然语言，人类可以追踪模型的决策过程，发现错误时也更容易定位

### 与 CoT、Action-only 的对比

| 方法 | 推理能力 | 实时信息 | 幻觉风险 | 透明度 |
|------|---------|---------|---------|-------|
| CoT | ✓强 | ✗无 | ✗高 | ✓好 |
| Action-only | ✗弱 | ✓有 | ✓低 | ✗差 |
| **ReAct** | ✓强 | ✓有 | ✓低 | ✓好 |

## 结论

1. 在 **HotpotQA**（多跳知识问答）上，ReAct 比 CoT 高约15%，比 Action-only 高约10%
2. 在 **FEVER**（事实验证）上，ReAct 显著减少了幻觉（因为有实际搜索结果支持）
3. 在 **ALFWorld**（文本游戏）上，ReAct 成功率比 Action-only 高约34%
4. ReAct 产生的推理链可以被人类监督，出错时可以人工干预（论文称之为"Human-in-the-Loop"能力）
5. ReAct 成为 **AI Agent** 的核心设计模式，LangChain、AutoGPT、ChatGPT Plugins 等都基于类似的"推理-行动"交织框架

---

## 局限性

1. **成本高**：每次完成一个任务需要多轮推理+工具调用，API 调用费用和时延都高
2. **错误传播**：如果某一步推理出错（如搜索到错误信息），后续所有步骤都会基于错误假设，可能一错再错
3. **工具质量依赖**：搜索引擎返回的结果不一定准确，模型无法判断工具结果的可靠性
4. **上下文窗口限制**：多轮思考-行动-观察循环会迅速填满模型的上下文窗口，限制了任务的复杂度
5. **复杂计划能力仍不足**：对于需要提前规划很多步骤的任务（如长期项目管理），ReAct 仍然容易陷入局部最优

---

## 一句话总结

ReAct 让语言模型在回答问题时交替进行"自然语言推理"和"工具调用"，实现了"想一想、查一查、再想想、再查查"的智能循环，既保留了思维链的推理能力，又通过实时工具调用消除了幻觉，成为现代 AI Agent（如 ChatGPT Plugins、LangChain Agent）的核心设计范式。
