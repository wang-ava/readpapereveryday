# CoCoMUT: Code Context Mining and Automated Dataset Generation

这篇面向 AI4S 的工作聚焦工程实践中最容易出问题的一环：方法级上下文的可复现实体化采集。它把零散、手工、难复现的数据预处理问题抽象成统一 pipeline，明显降低了软件工程任务的前置噪音。

对想把 LLM Agent 用在代码分析、文档生成、缺陷定位的团队来说，标准化上下文矿工比单一模型创新更能直接提升工程效率和可复现性。

- **核心问题**：软件工程领域的 LLM 工作流常需方法级上下文（调用关系、文档、类型层次等），但目前上下文采集依赖手工脚本，导致结果不可复现、跨仓库不一致。

- **方法概要**：
  1. 开发 CoCoMUT 工具链，支持 method/class/package/system 三种粒度数据输出。
  2. 自动发现工程结构与 build/classpath，构建 SootUp 静态调用图。
  3. 对齐字节码层 call edges 与 Spoon 源码抽取结果，生成可追踪的统一上下文。
  4. 输出版本化 JSON 与上下文数据集，可直接喂给文档生成、修复、测试、搜索等流程。

- **主要贡献**：
  1. 把上下文采集流程标准化为可复现实验流水线。
  2. 提供统一接口输出可组合的 task-agnostic 代码上下文。
  3. 给出规模化验证（20 个 Java 项目）与可复现性指标（97.8%、99.0%）。

- **关键实验或结果**：
  - 在 20 个真实 Java 项目（Maven/Gradle 各 10）上完成端到端处理，产出 56,512 条方法上下文记录与 386,048 条序列化 call edge。
  - 源码目标可对齐率 97.8%。
  - 随机抽样 200 条方法人工审计，99.0% 通过可适用正确性校验。

- **适合关注的原因**：
  1. 对研发流程的产出质量最直接影响之一就是上下文一致性，本工具显式解决这个瓶颈。
  2. 数据可版本化与任务无关，适配文档、测试、修复等多类 AI4S 场景。
  3. 工具链视角减少“模型改进但数据不可复现”的常见工程断层。

- **局限性或待验证点**：
  1. 当前重点覆盖 Java 生态，跨语言泛化仍需确认。
  2. 对超大单体仓库的处理成本与内存占用未在摘要中细化。
  3. 对动态语言特性（反射、元编程）场景的 context 完整性仍需专项测试。

- **对后续研究/应用的启发**：
  可作为 AI4S 工作流的基础设施层：先标准化上下文，再对模型做因果归因与评估，减少实验不可复现导致的误判风险。

- **一句适合 Obsidian 快速浏览的中文总结**：CoCoMUT 用统一的代码上下文提取与版本化数据集输出，直接增强 AI4S 中实验可复现性和方法间可比性。

## 标准化研究框架
- **Research question：** 在大规模软件项目中，能否通过统一的上下文挖掘+数据集生成流程提升代码智能任务的可复现性与可用性？
- **Literature：** 先前研究强调单任务抽取脚本，本文补充了“工程化统一采集”的系统化思路，接近 MLOps/数据工程范式。
- **Theory：** 在非社会科学研究中，这里对应“数据生成即实验设计假设”：通过统一语义上下文定义减少抽样和特征偏差。
- **Hypotheses：** 标准化 pipeline 可显著提升跨项目一致性和人工审计通过率，并降低下游任务的输入噪声。
- **Method：** 自动发现工程结构与构建信息，整合 SootUp 调用图与 Spoon 源码抽取，生成多粒度上下文与版本化 JSON 数据。
- **Data and Analysis：** 用 20 个真实 Java 仓库验证，指标为处理率、上下文记录规模、call edge 对齐率、人工抽样正确率。
- **Findings：** 本文报告显示高比例对齐率与高通过率，证明该管线在数据质量控制上的有效性。
- **Conclusion：** 对 AI4S 非社会科学研究，该字段可解读为“实验数据有效性框架”；结果支持“先标准化数据流程再谈模型创新”的工程优先顺序。
