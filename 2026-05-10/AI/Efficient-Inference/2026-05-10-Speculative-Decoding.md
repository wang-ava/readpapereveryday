# 2026-05-10 | Speculative Decoding: 用小模型加速大模型推理

> **发表**: ICML 2023  
> **机构**: Google  
> **作者**: Yaniv Leviathan, Matan Kalman, Yossi Matias  
> **领域**: 大语言模型推理加速 / 高效推理

## 背景知识：为什么大模型推理慢？

### 自回归生成的瓶颈

大语言模型每次生成一个 token 时，都需要完整地跑一次前向传播（Forward Pass）：

```
GPT-4（比如1750亿参数）生成每个词：
1. 把所有输入（之前所有词）传入模型
2. 模型所有参数都参与计算（1750亿次乘法！）
3. 输出一个词的概率分布
4. 采样一个词作为输出
5. 把这个词加入输入，回到步骤1...
```

这个过程不能并行——每一步都必须等前一步完成。结果：生成100个词就要跑100次前向传播。

**问题的本质**：大模型推理是**内存带宽瓶颈**，而不是计算瓶颈。

什么意思？GPU 的计算核（Compute Cores）很快，但每次生成一个词，都需要把模型所有参数（比如几百GB）从 HBM 内存读入计算单元。这个"读取"操作比计算本身慢得多。

### 关键洞察：验证比生成快！

这里有一个非常聪明的观察：

**生成 K 个 token 需要 K 次顺序的前向传播**  
**但验证 K 个 token，只需要 1 次并行的前向传播！**

为什么验证更快？
- 验证时，K 个候选 token 可以同时传入模型（一个完整的序列）
- 模型可以用注意力机制同时处理所有 K 个位置
- 这是 Transformer 天然支持的并行处理方式

**Speculative Decoding 的核心思路**：用一个小模型猜测多个词，再让大模型批量验证，利用验证快于生成的特性加速推理。

---

## 这篇论文干了什么？

**Speculative Decoding（推测解码）** 提出了一种无需任何质量损失的加速方法：

1. **小模型（Draft Model）猜测** γ 个 token
2. **大模型（Target Model）一次并行验证** γ 个 token
3. **接受/拒绝机制**：保证输出分布与原始大模型完全一致
4. **实测 2-3x 加速**，且输出质量与原大模型**完全一样**（数学保证）

---

## 怎么做的？

### 整体流程

**设 γ = 4**（即小模型每次猜4个词）

**步骤1：小模型猜测（Draft Phase）**
```
现有上下文：[你好，我是]
小模型猜测4个词：[人工，智能，助，手]
```
注意：这4次猜测是顺序的（小模型也是自回归的）。但因为小模型小，4次猜测总时间约等于大模型1次前向传播的时间。

**步骤2：大模型并行验证（Verification Phase）**
```
把 [你好，我是，人工，智能，助，手] 作为输入
大模型一次前向传播，输出序列中每个位置的下一词概率：
  位置0("你好") → P_target("我"的概率) = 0.85
  位置1("我") → P_target("是"的概率) = 0.90
  位置2("是") → P_target("人工"的概率) = 0.72
  位置3("人工") → P_target("智能"的概率) = 0.88
  位置4("智能") → P_target("助"的概率) = 0.60
  ...
```

**步骤3：接受/拒绝（Token-by-Token）**

对小模型猜测的每个词，根据大模型的概率决定接受还是拒绝：

```
对词 "人工"（位置2）：
- 小模型概率：P_draft("人工") = 0.80
- 大模型概率：P_target("人工") = 0.72

接受概率 = min(1, P_target / P_draft) = min(1, 0.72/0.80) = 0.90

以 90% 概率接受这个词，10% 概率拒绝
```

**为什么要这样计算？**

这个接受/拒绝机制确保了：无论接受还是拒绝，最终的词分布都和直接用大模型采样的结果**完全一样**！这是论文中严格证明的。

**步骤4：处理拒绝**

如果某个词被拒绝（比如"助"被拒绝了）：
- 停止，不继续接受后面的词
- 从大模型的（修正后的）分布中重新采样这个位置的词
- 从这个新词继续生成

### 一次迭代的时间分析

```
传统自回归（生成1个词）：
  1次大模型前向传播 ← 受内存带宽限制

Speculative Decoding（目标：生成γ个词）：
  小模型：γ次小模型前向传播（约等于1次大模型前向传播的时间）
  大模型：1次大模型前向传播（验证γ个位置）

如果γ个词都被接受：速度提升 ≈ γ倍
如果平均接受了α个词：速度提升 ≈ α/(1+α/γ×(大/小速度比)) 
```

关键：**只要小模型和大模型的输出有足够的重叠（高接受率），就会有显著加速。**

## 主要结论

### 在 Chinchilla（70B）上的实验

使用 10M 参数的小模型作为 draft model，在文本生成任务上：

| 指标 | 传统方法 | Speculative Decoding |
|------|---------|---------------------|
| 吞吐量 | 基线 | **2-3x** |
| 输出质量 | 基线 | **完全相同**（数学证明）|
| 内存增量 | - | 很小（加载小模型）|

### 接受率分析

- 代码生成任务：接受率高（≈80%），加速效果好
- 创意写作任务：接受率较低（≈50-60%），加速效果适中
- 小模型越与大模型"志同道合"，接受率越高，加速越好

---

## 局限性

1. **需要小模型与大模型同系列**：如果小模型和大模型分布差异太大，接受率低，加速效果差。理想情况是同一个家族（如 LLaMA-7B 作为 LLaMA-70B 的 draft model）
2. **需要同时加载两个模型**：增加了显存需求（虽然小模型很小）
3. **对 prefill 没有帮助**：加速主要在"decoding（逐词生成）"阶段，"prefill（处理长输入）"没有改善
4. **实现复杂**：需要定制化的批量接受/拒绝逻辑，不能直接用现有框架
5. **低接受率时不如直接用大模型**：如果小模型猜的都是错的，反而增加了开销

---

## 为什么值得关注？

Speculative Decoding 是一个优雅的**无损加速**技术：

1. **数学上严格**：不是"可能差一点点"，而是**保证输出分布完全相同**
2. **被行业广泛采用**：Google（Gemini）、Anthropic（Claude）、Meta 等都在使用此技术或其变体
3. **框架兼容**：vLLM 等推理框架已内置支持

类比：就像投机的预测——你先"赌一把"，猜测对了就赚到了，猜错了也没有损失（因为修正机制保证了正确性）。

后续工作还提出了更多变体：
- **SpecTr**（谱变换体）：更优化的 draft 策略
- **Lookahead Decoding**：不需要小模型
- **Medusa**：给大模型增加多个"猜测头"，内置于模型本身
