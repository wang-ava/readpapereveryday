## 背景知识：为什么 LLM 推理这么慢？这么贵？

### LLM 生成文字的过程：逐词生成

大语言模型生成文字是**自回归**的——每次只生成一个词（token），然后把这个词加入上下文，再预测下一个词：

```
输入："你好"
第1步：模型预测 → "我"  
第2步：模型预测 → "是"（基于"你好，我"）
第3步：模型预测 → "AI"（基于"你好，我是"）
...
```

每一步都需要一次完整的前向传播，不能并行，所以很慢。

### KV Cache：加速但耗内存

为了避免重复计算，Transformer 模型使用 **KV Cache（Key-Value 缓存）**：

每次生成新词时，不重新计算之前所有词的注意力键值（Key-Value），而是把它们缓存起来直接使用。

**问题**：KV Cache 占用大量 GPU 内存。

例如，处理一个长度为 2048 tokens 的请求，LLaMA-13B 的 KV Cache 需要 **~1.7 GB** 的 GPU 内存。

如果同时处理100个请求（批量推理），就需要 **~170 GB**——远超单张 A100 的 80 GB 显存！

### 传统内存管理的问题：浪费严重！

在 vLLM 之前，KV Cache 是这样分配的：

```
分配内存时：预先分配 MAX_LENGTH 的连续内存块
（比如，每个请求分配能放 2048 个token的连续空间）

实际使用时：请求A可能只用了500个token
  [AAAA....空...空...空...空] ← 1548个位置是空的！

请求B请求内存时：虽然有1548个空位，但它们不连续，
所以系统无法分配给请求B！
```

这导致了严重的内存碎片化：

**内存利用率只有 20-40%**——也就是说，60-80%的 GPU 内存被白白浪费！

## 这篇论文干了什么？

**vLLM（Virtual LLM）** 提出了 **PagedAttention**，用操作系统中"虚拟内存分页"的思想彻底解决了 KV Cache 的内存浪费问题。

核心贡献：
1. **PagedAttention**：把 KV Cache 分成固定大小的"页"（Block），按需分配
2. **内存利用率接近100%**（vs. 传统的 20-40%）
3. **2-4x 推理吞吐量提升**，无需任何质量损失
4. **成为 LLM 推理的行业标准**：OpenAI、Anthropic、Google 等都在内部使用类似技术

## 怎么做的？

### 关键思路：借鉴操作系统的虚拟内存

操作系统管理物理内存的方式非常聪明：

- **虚拟内存**：程序看到的是连续的虚拟地址空间（比如 0 到 4GB）
- **物理内存分页**：实际内存分成固定大小的"页"（Page）
- **页表**：维护虚拟地址到物理地址的映射
- **按需分配**：只有当程序真正访问某个虚拟地址时，才分配对应的物理页

**PagedAttention 做了完全类似的事**，只是把"程序内存"换成了"KV Cache"：

| 操作系统概念 | PagedAttention 对应 |
|------------|-------------------|
| 物理内存页（Page）| KV Cache Block（一组连续token的KV） |
| 虚拟内存地址 | 请求中的 token 序列位置 |
| 页表 | Block 表（Block Table）|
| 按需分配 | 请求生成时按需分配 Block |

### PagedAttention 的工作原理

**Step 1：把 KV Cache 分成固定大小的 Block**

```
每个 Block = 16 个 token 的 KV Cache
（Block 大小可配置，16是常见选择）
```

**Step 2：为每个请求维护 Block 表（Block Table）**

```
请求 A 的 Block 表：
序列位置 0-15  → 物理 Block #7
序列位置 16-31 → 物理 Block #2
序列位置 32-47 → 物理 Block #15
...
```

物理 Block 不需要连续！可以"东拼西凑"地分配内存中任何空闲的 Block。

**Step 3：注意力计算时按 Block 访问**

```python
# 伪代码
for block in request.block_table:
    k_block = get_key_cache(block)  # 从物理 Block 读取
    v_block = get_value_cache(block)
    attention_scores += dot(query, k_block)
    # ...
```

**Step 4：内存管理器统一管理所有 Block**

```
可用 Block 池（类似操作系统的物理内存池）：
[Block #1 空闲] [Block #2 占用by A] [Block #3 空闲] ...

请求 A 新增一个 token → 从空闲池取 Block #1 → A 的 Block 表增加一项
请求 A 完成 → 释放 Block #1、#2 回空闲池
```

### 内存浪费从哪里来的？对比分析

**传统方式**：
```
为请求 A 预分配 2048 个 token 的连续空间
A 实际只用了 500 个 token
浪费：1548 个 token 的空间（76% 浪费）
```

**PagedAttention**：
```
A 生成 500 个 token，使用了 500/16 ≈ 32 个 Block
精确地只占用 32 个 Block 的空间
内存浪费：最多 15 个 token（1个 Block 内部）= < 3% 浪费
```

### Bonus：KV Cache 共享（Copy-on-Write）

PagedAttention 还支持**在多个请求之间共享 KV Cache**：

**场景：Beam Search**
- 模型同时追踪多个生成候选（比如 beam_size=4）
- 前缀部分是完全相同的
- 传统方式：每个 beam 存储完整的前缀 KV Cache（4份重复）
- PagedAttention：前缀 Block 共享，只在内容分叉时复制（Copy-on-Write）

**场景：System Prompt 共享**
- 所有请求都有相同的系统提示（如 "You are a helpful assistant"）
- 系统提示的 KV Cache 只存一份，所有请求共享

|------|-------------------|---------|
| HuggingFace Transformers | 基线 | 1x |

（在 A100 GPU 上，LLaMA-13B 模型，16个并发请求）

### 内存利用率

| 系统 | KV Cache 内存利用率 |
|------|-------------------|
| 传统方法 | 20-40% |
| **vLLM** | **> 96%** |

几乎消除了内存浪费。

---

## 局限性

1. **Block 大小需要调优**：Block 太小 → 内存管理开销大；Block 太大 → 小请求也浪费空间
2. **PagedAttention 的硬件实现复杂**：需要定制 CUDA kernel，实现难度高
3. **不加速首个 token 生成（prefill）**：PagedAttention 主要优化的是"生成新 token（decoding）"阶段，对"处理输入（prefill）"阶段改善有限
4. **与某些注意力优化冲突**：与 FlashAttention 的完全融合实现有冲突，需要特殊处理
